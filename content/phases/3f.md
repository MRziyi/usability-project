# Phase 3F — Final Website Usability Makeover Redesigns

This page contains our **final redesign decisions** for Google Looker Studio, incorporating peer feedback from Phase 3D/3E and a fresh re-review of the 3C work.

---

## 1) Design Evolution: 3C → 3F

### What stayed the same

Our core problem set (P1–P8) and the three design principles from 3C remain:

- Discoverability over "hidden power"
- Make system state legible
- Align with dominant mental models (Docs / Slides / PowerPoint)

Peer reviewers confirmed the problems were real — no new critical issues were surfaced, which validates our original analysis.

### What changed (summary)

| Problem | 3C direction | 3F change | Driver |
|---|---|---|---|
| P1 Data Import | Option A preferred | Option A + first-time guidance layer | Peer vote confirmed A; Thiri's case for Option C scaffolding |
| P2 Dataset binding | Option A (pill) or Option B (banner), undecided | Both: collapsible pills + post-upload banner | Giang/Zixiao/Cindy clutter concern; Dua's preference for banner |
| P4 Chart editing | Option A (decomposed field rows) | Option A + progressive disclosure ("Advanced ▾") | Dua's suggestion; also addresses Saira's beginner/expert concern |
| P6 Annotation | Option A (handles + move/edit clarity) | Option A + double-click to enter edit mode | Thiri's Slides-style suggestion |
| P3, P5, P7, P8 | Multiple options listed | Unchanged — 3C directions confirmed | No new feedback; deprioritized |

## 2) Final Redesign Decisions

### P1 — Data Import

**Final decision: Option A + first-time guidance layer**

The primary "Add data" modal shows a large, first-class **Upload file (CSV/Excel)** button above the connector list — prominent, with an upload icon, labeled "Best for local files." For **first-time users only**, a lightweight contextual hint appears anchored to the Upload button, walking through the three steps: *Choose source → Confirm schema → Apply to report.* A "Skip guide" link dismisses it permanently.

**Why not pure Option A:** Thiri's argument for Option C (Import Wizard) is valid — stronger scaffolding catches more first-time errors. The guidance layer captures that benefit on first use without adding permanent steps for returning users.

**Why not pure Option C:** The full wizard adds navigation overhead every time unless opt-out is carefully designed. Option A + guidance layer achieves the same goal with less friction.

**Preferred over alternatives:** Option B (guided connector screen) improves learnability incrementally but still requires understanding the connector model. Not as direct as Option A for local file users.

### P2 — Dataset-to-Chart Binding

**Final decision: Collapsible color-coded data pills (Option A refined) + post-upload banner (Option B)**

Two complementary changes that address different moments in the workflow:

**Data pills (ongoing reference):** Each chart shows a small color-coded badge. Default state: a colored dot only (minimal, low noise). Hover or entering edit mode expands the badge to a full label (*worldcities ▾*) with a one-click dataset quick-switch. Color coding makes it immediately scannable — charts sharing the same dataset share the same color.

**Post-upload banner (one-time intervention):** After a dataset is connected, a non-blocking banner appears at the top of the canvas: *"Dataset added — worldcities.csv. Apply to charts? [Apply all] [Choose…] [Dismiss]."* This prevents the most common failure mode (users believing upload = automatically applied) at the exact moment it can occur.

**Why both:** The banner solves the initial confusion; the pills solve the ongoing "which chart uses which data?" question during editing. They are additive, not redundant.

**Trade-off managed:** Pills are minimal by default (color dot only) — directly addressing the visual clutter concern raised independently by Giang, Zixiao, and Cindy.

### P4 — Chart Editing Controls

**Final decision: Option A (decomposed field rows) + progressive disclosure**

The editing panel is restructured in two layers:

**Layer 1 — Always visible (beginner default):**
- Chart type picker (always expanded, not collapsed)
- Dimension and Metric field rows with labeled inline actions: *[Change] [Edit] [Info]*
- Sort control with direction toggle

**Layer 2 — Behind "Advanced ▾" toggle:**
- Data blending
- Custom metric expressions
- Filter conditions
- Data type overrides

**Why progressive disclosure instead of Saira's dual-mode UI:** A full dual-mode (two separate layouts at login) has high implementation cost and maintenance overhead, and forces users to self-classify as "beginner" or "expert" before they know what they need. Progressive disclosure achieves the same separation within a single interface — beginners see a clean panel by default; experts click once to reveal everything. No mode switch, no settings navigation, no second design to maintain.

**Why the panel over right-click (Option B):** Multiple reviewers and Thiri specifically confirmed that the panel redesign is more discoverable than right-click. Right-click remains a valid power-user shortcut layer but should not be the primary path to common actions.

### P6 — Text & Annotation Authoring

**Final decision: Option A enhanced with double-click edit mode**

Three concrete changes:

1. **Larger, clearer handles:** Bounding box visible on selection; 8 resize handles at corners and midpoints (matching Slides/PowerPoint standard).
2. **Forgiving drag target:** Dragging anywhere on the bounding box frame moves the element; only the interior activates text editing.
3. **Double-click to enter edit mode:** Single click = select and move; double-click = enter text editing. This is the same interaction model as Google Slides, Keynote, and Figma, and directly prevents the current issue where clicking to reposition a text box accidentally places the cursor and disrupts formatting.

**Why add double-click:** Thiri specifically raised this pattern, and it is well-established across every major presentation and design tool. It makes the "move vs edit" distinction explicit through interaction depth rather than a visible mode toggle, keeping the UI clean.

**Preferred over Option B (Annotation Mode toggle):** Mode toggles add mental overhead and can trap users who forget which mode they are in. The double-click interaction achieves clear mode separation without a persistent UI element.

**Preferred over Option C (explain Responsive constraints):** Option C is a patch on a confusing system; this change fixes the underlying interaction model.

![P6 Final — Enhanced handles with double-click edit mode](/media/3f/p6-final.png)

### P3, P5, P7, P8 — Confirmed from 3C

No peer feedback surfaced new concerns for these problems. 3C directions are confirmed:

- **P3** (Discoverability): Feature Templates near "Create/Blank report"; show "Start from template?" suggestion on blank report creation.
- **P5** (Layout legibility): When truncation occurs, show inline explanation + "Fix it" action in the style panel.
- **P7** (Reliability): Autosave + visible save status; crash recovery with restore-last-state prompt.
- **P8** (Paper-cuts): Auto-select title field on first click; prevent mid-text cursor placement in critical fields.

## 3) Design Rationale: Full Evolution

**How we got from 3C to 3F — four moves:**

| Move | What changed | Evidence / Reason |
|---|---|---|
| P1: A + first-time guidance | Added contextual first-use tooltip to Option A | 6 reviewers confirmed A; Thiri's Option C argument valid for first use only |
| P2: Pills + banner combined | Adopted both instead of choosing one | They solve different moments; pills → ongoing reference; banner → one-time intervention |
| P4: Progressive disclosure added | Layered "Advanced ▾" onto decomposed field rows | Dua's suggestion; resolves beginner/expert tension without dual-mode cost |
| P6: Double-click edit mode | Added explicit interaction depth to Option A | Thiri's suggestion; matches established pattern across Slides/Keynote/Figma |

**Alternate solutions still in play:**

- P1: Pure Option C (full Import Wizard) remains valid for a product serving mostly first-time-only users (e.g., classroom or onboarding context).
- P4: Right-click context menu (Option B) remains a valid complementary layer for power users — it does not conflict with the panel redesign.
- P2: Option C (smart apply + undo) is the most automated approach and may suit a product team willing to invest in dataset-matching heuristics.

**Our top recommendation across all four:** The progressive disclosure pattern (P4) has the highest leverage — it applies a single consistent principle to all panel sections, not just field rows, and directly addresses the most frequent breakdown category in our user studies.

---

## 4) Reflection

Our original problem analysis held up well under peer review — reviewers with and without Looker Studio experience confirmed the same friction points. The feedback most meaningfully shaped **P2** (where we resolved the "pick one option" ambiguity into a two-layer solution) and **P4** (where Dua's progressive disclosure suggestion gave us a clean answer to the beginner/expert tension that we had framed as a trade-off but not resolved).

The suggestion we found most interesting but did not adopt — Saira's dual-mode UI — points toward a real long-term product strategy question: at what scale does it make sense to invest in distinct learning curves for different user types? Our answer here was "not at this scope," but the question is worth revisiting for a production team.

If we continued: validate the P4 progressive disclosure layout and P6 double-click model with 2–3 quick usability tests, and refine the color-coding logic for P2 pills (how colors are assigned, whether they persist across sessions).
