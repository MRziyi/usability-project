# Phase 3E — Group Reply to Peer Feedback

We received nine comments during Phase 3D and posted a summary reply on Canvas on April 21 (Miantong Zhang on behalf of Group 5). This page reproduces that reply and documents our design decisions in response to the feedback.

---

## Summary Reply (Posted to Canvas, April 21)

> Hi everyone,
>
> Thank you all for taking the time to look through our Google Looker Studio redesign project and leave helpful comments. We really appreciate all of the feedback and suggestions.
>
> We are glad that many of you found our website clear, organized, and easy to follow. We spent a lot of time trying to make the Problem Inventory easy to understand, so it was great to hear that elements like severity ratings, rationale, and redesign comparisons were useful.
>
> We also noticed that several people preferred **Option A for our P1 Data Import redesign**, especially the clear "Upload File" entry point. This matches what we found in our user study, where new users had trouble figuring out how to upload local files. Because of that, we will continue developing that idea in our final redesign.
>
> Some comments also mentioned that added labels, pills, or indicators could **create clutter in more complex dashboards**. We think that is a very good point, and we will look into ways to make those elements collapsible or only visible when needed.
>
> We also appreciated suggestions about beginner vs experienced users, improving comparisons with the original interface, and simplifying editing tools. These ideas will help us improve the next version of our project.
>
> Thanks again for all the thoughtful feedback. It was very helpful for us, and we will use it as we continue working on the final redesign.

## Our Decisions

### Accepted: Collapsible, color-coded data pills (P2)

Giang, Zixiao, and Cindy independently noted that persistent data labels could make dense dashboards visually noisy. We agree. In the final redesign, data pills are **minimal by default** (a small color dot per chart), expanding on hover or in edit mode to show the full label and a quick dataset-switch control. Color coding makes it instantly scannable when multiple datasets are in use.

### Accepted: Post-upload banner (P2 Option B)

Dua's argument is sound — the banner addresses the misunderstanding at exactly the moment it occurs. We adopt Option B alongside the refined pills: the banner fires once after upload; the pills provide ongoing reference while editing.

### Accepted: Progressive disclosure in chart editing panel (P4)

Dua's suggestion to show basic controls by default and hide advanced options behind a toggle is a well-established usability pattern. We combine this with our existing Option A (decomposed field rows): the panel defaults to the most common actions (Change field, Chart type, Sort), with an expandable "Advanced ▾" section for power features. This reduces cognitive load for new users without removing expert-level access.

### Partial accept: Saira's dual-mode UI (beginner / expert)

Saira proposed two separate UI layouts selectable at login. This is a thoughtful architectural idea — but full dual-mode carries significant implementation cost, maintenance overhead, and risks fragmenting the user's mental model when switching. We address the same underlying need more lightly: **progressive disclosure in P4** effectively creates a beginner-friendly default view without requiring a mode switch or self-classification. Users who want more simply expand "Advanced ▾". We note Saira's direction as a sound long-term vision for a large product team, but out of scope here.

### Partial accept: Thiri's preference for Option C (P1 Import Wizard)

Thiri made a compelling case that an interface-embedded wizard provides better first-time scaffolding than a standalone upload button. We agree with the benefit — but a full wizard adds steps on every subsequent use unless opt-out is carefully designed. Our final decision: **Option A as the primary entry point, with a lightweight first-time guidance layer** — a contextual tooltip on first visit that walks through the three-step flow (choose source → confirm schema → apply to report). First-time users get Option C's scaffolding; returning users see a clean, uninterrupted modal.

### Rejected: Adding original interface screenshots (Jingyuan)

Jingyuan suggested including screenshots of the current Looker Studio UI alongside each redesign to make the "before" state clearer. We agree this would strengthen the evidence. However, capturing and annotating accurate before-screenshots for all eight problems is not feasible within our remaining timeline. Our problem descriptions and evidence signals in the Problem Inventory already establish the current state. We note this as ideal future work.

---

## Impact on Phase 3F

The accepted feedback drives four concrete changes in the final redesigns:

1. **P1**: Option A + first-time guidance layer (combines strengths of A and C)
2. **P2**: Collapsible color-coded data pills (Option A refined) + post-upload banner (Option B)
3. **P4**: Decomposed field rows (Option A) + progressive disclosure ("Advanced ▾" toggle)
4. **P6**: Option A enhanced with double-click to enter edit mode
